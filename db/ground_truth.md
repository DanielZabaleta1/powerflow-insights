# Ground truth queries — demo dataset

Hand-written and hand-verified against `demo.leads` / `demo.activities` after
seeding (500 leads, 1918 activities, generated by `db/generate.ts` with a
fixed RNG seed — see there for the exact conversion model). These are the
source of truth for two things: confirming the synthetic funnel is realistic,
and the few-shot examples baked into the NL→SQL prompt in Fase 4.

Run against Postgres 17 on the Power Flow OS Supabase project, `demo` schema.

## 1. Global funnel (% of total leads reaching each stage)

```sql
select
  count(*) as total_leads,
  round(100.0 * count(*) filter (where status not in ('New','Ready')) / count(*), 1) as pct_contacted,
  round(100.0 * count(*) filter (where status in ('Replied','Call booked','Proposal sent','Won','Lost')) / count(*), 1) as pct_replied,
  round(100.0 * (select count(distinct lead_id) from demo.activities where type = 'call_booked') / count(*), 1) as pct_call_booked,
  round(100.0 * (select count(distinct lead_id) from demo.activities where type = 'proposal_sent') / count(*), 1) as pct_proposal,
  round(100.0 * count(*) filter (where status = 'Won') / count(*), 1) as pct_won
from demo.leads;
```

**Result:**

| total_leads | pct_contacted | pct_replied | pct_call_booked | pct_proposal | pct_won |
|---|---|---|---|---|---|
| 500 | 75.0% | 18.6% | 7.2% | 4.8% | 3.6% |

Target from the project guide was ~75 / 22 / 8 / 5 / 2 — all five land within
a few points of target given a 500-row sample. Won came in a bit hot (3.6%
vs. 2% target) because `warm`/`referral` leads close disproportionately
often; see the per-channel breakdown below for why.

## 2. Conversion by channel — the "referral/warm convert ~3x better than cold LinkedIn" claim

```sql
select
  channel,
  count(*) as total,
  round(100.0 * count(*) filter (where status not in ('New','Ready')) / count(*), 1) as pct_contacted,
  round(100.0 * count(*) filter (where status in ('Replied','Call booked','Proposal sent','Won','Lost')) / count(*), 1) as pct_replied_or_further,
  round(100.0 * count(*) filter (where status = 'Won') / count(*), 1) as pct_won
from demo.leads
group by channel
order by pct_replied_or_further desc;
```

**Result:**

| channel | total | pct_contacted | pct_replied_or_further | pct_won |
|---|---|---|---|---|
| warm | 40 | 72.5% | 40.0% | 25.0% |
| referral | 24 | 75.0% | 33.3% | 16.7% |
| agency | 67 | 74.6% | 19.4% | 3.0% |
| upwork | 133 | 72.2% | 17.3% | 1.5% |
| linkedin | 236 | 77.1% | 11.4% | 0.0% |

warm/linkedin = 40.0 / 11.4 = **3.5x**. referral/linkedin = 33.3 / 11.4 =
**2.9x**. Both land on the "~3x" target. linkedin's 0.0% won is expected, not
a bug — the model's implied won-rate for linkedin is ~0.4% (0.76 contacted ×
0.19 replied × 0.24 call-booked × 0.41 proposal × 0.26 won), so 0 wins out of
236 leads is well within normal sampling variance, and is itself a realistic
talking point (cold outreach alone rarely closes without a warm signal).

## 3. Weekly time series (for the "how many leads by week" suggested question)

```sql
select date_trunc('week', created_at)::date as week_start, count(*) as leads_created
from demo.leads
group by 1
order by 1;
```

**Result (26 weeks):** ranges from 10 to 29 leads/week, no zero weeks, mean
~19.2/week — consistent with the weekend-downweighted daily generation.
Full row-by-row output isn't reproduced here (26 rows); re-run the query
against `demo.leads` to see it live.

Weekend seasonality, checked separately:

```sql
select extract(dow from created_at) as day_of_week, to_char(created_at,'Dy') as day_name, count(*)
from demo.leads group by 1, 2 order by 1;
```

Sat+Sun average 30.5 leads/day vs. 87.8/day on weekdays — a 0.35 ratio,
matching the 0.4 weekend weight built into `sampleCreatedAt()` in
`db/generate.ts`.

## 4. Average time from first contact to reply

```sql
select
  round(avg(extract(epoch from (r.occurred_at - m.occurred_at)) / 86400.0), 1) as avg_days_contact_to_reply,
  count(*) as replies_measured
from demo.activities r
join demo.activities m on m.lead_id = r.lead_id and m.type = 'message_sent'
where r.type = 'reply_received';
```

**Result:** 6.0 days average, over 93 measured replies. This is higher than
the "1-5 days after the touch that triggered it" range in the generator
because it's measured from the *first* `message_sent`, not the touch that
actually got the reply — a lead that replies after Touch 3 (`+8d` from first
message) plus a 1-5 day reply lag will show up here as 9-13 days. Both
framings are legitimate; this one matches how a business user would actually
ask the question ("how long from first outreach to reply").
